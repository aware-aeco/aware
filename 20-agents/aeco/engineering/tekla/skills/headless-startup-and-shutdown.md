# Tekla headless startup + shutdown

Tekla Structures has **no headless / server / Docker build** — but it CAN be launched without manual interaction by pre-supplying every dialog answer via a `Bypass.ini` file. This is the foundation of any automated test harness that exercises real Tekla.

## Canonical launch command

```
TeklaStructures.exe -I "<Bypass.ini>" "<model_path>"
```

Verified working against Tekla 2026.0 (May 2026). Reaches Open API readiness in **~30 seconds** from process start with a small / blank model. Larger production models will take longer.

## `TeklaStructures.exe` command-line parameters (full reference)

From [Trimble's official 2026 documentation](https://support.tekla.com/doc/tekla-structures/2026/cus_create_startup_shortcuts_with_customized_initializations):

| Parameter | Description |
|---|---|
| `-I <ini_file>` (capital) | Load .ini BEFORE environment .ini files. Can specify multiple times. **This is the bypass-the-setup-dialog hook.** |
| `-i <ini_file>` (lowercase) | Load .ini AFTER role .ini files. Can specify multiple times. For per-project/per-user overrides. |
| `<model_path>` | Open this model after startup. Quote it if it contains spaces. |
| `<model_path> /autosaved` | Open the autosave snapshot of the given model. |
| `/create:<model_path>` | Create a new (blank) model and open it. |
| `/create:<model_path> /modelTemplate:<template_name>` | Create from a model template (e.g. `Cast-in-Place`). |
| `/create:<model_path> /server:<host:port>` | Create a new multi-user model on the given server. |
| `-m <macro_file>` | Run a macro (`.cs`) after startup. Combine with `-I` + model path for fully scripted runs. |

**Maximum shortcut length: 256 chars.** For long parameter sets, chain initialization files via `call` directives inside a master `.ini` instead.

## `Bypass.ini` — the dialog-skip schema

Located at `%XSDATADIR%\Environments\<env>\Bypass.ini` per install (default template ships with everything commented out). Required keys to skip the Tekla Structures - Setup dialog:

```ini
set XS_LICENSE_SERVER_HOST=https                              ; or "<port>@<host>" for on-prem
set XS_DEFAULT_LICENSE=Partner                                ; CARBON | GRAPHITE | DIAMOND | Partner | FULL
set XS_DEFAULT_ENVIRONMENT=%XSDATADIR%\Environments\<env>\env_<env>.ini
set XS_DEFAULT_ROLE=%XSDATADIR%\Environments\<env>\role_<role>.ini
```

### Special values

- **`XS_LICENSE_SERVER_HOST=https`** — magic string meaning "use the Trimble Identity online subscription path". Use this for cloud-license users. Old-style is `<port>@<hostname>` for on-prem floating-license servers.
- **`%XSDATADIR%`** — Tekla-set env var pointing at `C:\ProgramData\Trimble\Tekla Structures\<version>\`. Always available inside Bypass.ini, never expand it yourself.

### License-tier examples (`XS_DEFAULT_LICENSE`)

| Value | Edition |
|---|---|
| `CARBON` | Tekla Structures Carbon (entry tier) |
| `GRAPHITE` | Tekla Structures Graphite |
| `DIAMOND` | Tekla Structures Diamond (full / detailing) |
| `Partner` | Trimble Partner subscription |
| `FULL` | Legacy "full" perpetual license |

## Launch from a sidecar (C#)

```csharp
using System.Diagnostics;

void LaunchTeklaHeadless(string version, string modelPath, string bypassIniPath)
{
    var teklaExe = $@"C:\Program Files\Tekla Structures\{version}\bin\TeklaStructures.exe";
    var psi = new ProcessStartInfo
    {
        FileName        = teklaExe,
        Arguments       = $"-I \"{bypassIniPath}\" \"{modelPath}\"",
        WindowStyle     = ProcessWindowStyle.Maximized,    // visible status bar for visual verification
        UseShellExecute = true,                            // required for WindowStyle to honour Maximized
    };
    var p = Process.Start(psi);
    WaitForOpenApiReady(version, timeoutSeconds: 120);
}
```

`WindowStyle = Maximized` requires `UseShellExecute = true`. Without that, the OS silently ignores the window-style hint.

## Open API readiness — DO NOT trust process existence

**Process running ≠ Open API reachable.** Tekla's `Trimble.Remoting` IPC channel (`Tekla.Structures.Model-TeklaStructures-Console:<version>`) doesn't exist until Tekla finishes loading the model. Polling `Process.GetProcessByName("TeklaStructures")` returning a PID will say "ready" 3 seconds after launch — and your first Open API call will fail with `Trimble.Remoting.RemotingIOException: Cannot connect to remoting service` (wrapping a `System.IO.FileNotFoundException` from `MemoryMappedFile.OpenExisting`).

Empirically: **30 seconds is the typical floor; production models take 1-3 minutes.** Always poll for connection-readiness, never assume from process existence.

### Polling pattern

```csharp
bool WaitForOpenApiReady(string version, int timeoutSeconds)
{
    var deadline = DateTime.UtcNow.AddSeconds(timeoutSeconds);
    while (DateTime.UtcNow < deadline)
    {
        Thread.Sleep(5_000);
        try
        {
            var model = new Tekla.Structures.Model.Model();
            if (model.GetConnectionStatus()) return true;
        }
        catch
        {
            // Trimble.Remoting throws if the IPC channel isn't up yet — keep polling.
        }
    }
    return false;
}
```

`Model.GetConnectionStatus()` is the canonical readiness check. It returns `false` (or throws `RemotingIOException`) until Tekla has loaded the model AND registered the IPC channel.

## Shutdown — default is CLEAN, force is emergency only

**`Process.Kill` loses unsaved model edits.** Never default to force-kill — always go through Tekla's Open API event chain. Force-kill is reserved for: a) the model has no unsaved state we care about, b) Tekla is hung and the event chain won't complete, or c) test setups using throwaway `/create:` models where nothing exists worth saving.

### The full Tekla.Structures.Model.Events surface

Compiled from both the [official Tekla 2025 docs](https://developer.tekla.com/doc/tekla-structures/2025/events-class-52530) **and** runtime enumeration of Tekla 2026. The runtime list is a strict superset — 2026 adds several events not in the published 2025 reference. Below: complete table.

#### Lifecycle events (load / save / exit)

| Event | Delegate signature | When it fires | Notes |
|---|---|---|---|
| `ModelLoad` | `void()` | After a model loads | Pair with `WaitForOpenApiReady` polling for full readiness |
| `ModelLoadInfo` | `void(string)` | After model loads, with info string | Diagnostic |
| `ModelSave` | `void()` ⚠️ on 2026 (some older docs say `void(string)` — they're wrong for 2026) | After a save **completes** | **Does NOT fire if `Save()` was a no-op (model already clean)** |
| `ModelSaveInfo` | `void(string)` | After save, with info string | Diagnostic |
| `ModelSaveAs` | `void()` | After `Operation.SaveAs(path)` completes | Save-as path, not regular save |
| `ModelUnloading` | `void()` (async) | Before model unloads (asynchronous) | Close-imminent signal |
| `ModelUnloadingSync` | `void()` (sync) | Before model unloads (synchronous) | Use this for resource cleanup that needs the model still alive |
| `TeklaStructuresExit` | `void()` | Before Tekla Structures exits | **Authoritative app-exit signal.** Fires INSIDE Tekla just before process death — external sidecars receive it on the Trimble.Remoting channel right before the channel drops. |

#### Model-data events (changes)

| Event | Delegate signature | When it fires | Notes |
|---|---|---|---|
| `ModelChanged` | `void(List<ChangeData>)` | After model changes | Does NOT fire on undo/redo |
| `ModelObjectChanged` | `void(List<ChangeData>)` | After object changes | DOES fire on undo/redo (the difference vs `ModelChanged`) |
| `ProjectInfoChanged` | `void()` | After project info changes | Does NOT fire on undo/redo |
| `ModelObjectNumbered` | `void()` | When a model object is numbered | |
| `Numbering` | `void()` | After model numbering completes | Whole-model variant |
| `UndoClicked` | `void()` | When user clicks undo | |
| `Interrupted` | `void()` | When user interrupts an operation (Esc/Cancel) | |
| `CommandStatusChange` | `void()` | When a command starts or ends | |

#### Selection + UI events

| Event | Delegate signature | When it fires |
|---|---|---|
| `SelectionChange` | `void()` | User changes model object selection |
| `AnnotationSelectionChange` | `void()` | User changes annotation selection |
| `HiddenObjectsChanged` | `void()` | User hides an object, or view redraws |
| `TemporaryStatesChanged` | `void()` | Temporary visualisation states change |
| `ClipPlaneChanged` | `void()` | User deletes a clip plane |

#### View events

| Event | Delegate signature | When it fires |
|---|---|---|
| `ViewClosed` | `void()` | A view closes |
| `ViewOpened` | `void()` | A view opens (2026+, not in 2025 docs) |
| `ViewCreated` | `void()` | A view is created (2026+) |
| `ViewModified` | `void()` | A view is modified (2026+) |
| `ViewDeleted` | `void()` | A view is deleted (2026+) |
| `ViewCameraChanged` | `void()` | View camera changes |
| `ModelViewDrawn` | `void()` | Model view is drawn (2026+) |

#### Clash detection events

| Event | Delegate signature | When it fires |
|---|---|---|
| `ClashCheckDone` | `void()` | After clash check completes |
| `ClashDetected` | `void(ClashCheckData)` | After each individual clash is detected |

#### Misc events (mostly 2026 additions)

| Event | Delegate signature | When it fires |
|---|---|---|

### How to discover delegate signatures at runtime (the safe way)

**Trimble changes delegate signatures between versions.** Don't hardcode — reflect:

```csharp
var eventsType = modelAsm.GetType("Tekla.Structures.Model.Events")!;
foreach (var ev in eventsType.GetEvents())
{
    var invoke = ev.EventHandlerType!.GetMethod("Invoke")!;
    var paramTypes = invoke.GetParameters().Select(p => p.ParameterType.Name);
    Console.WriteLine($"{ev.Name}: ({string.Join(", ", paramTypes)}) -> {invoke.ReturnType.Name}");
}
```

Use `DynamicMethod` to emit a handler with the discovered shape (see the `Close` verb implementation in `cli-tekla/Program.cs` for the working pattern).

### Important: Tekla has NO external-process exit API

**Empirically verified by reflecting every Tekla.* assembly:** there is no public method on `Operation`, `Model`, `TeklaStructures`, or any other type that lets an external process tell Tekla to exit. `Process.CloseMainWindow()` (WM_CLOSE) is silently ignored. Trimble's design intent: the user closes Tekla; external code can save into it but can't shut it down.

This forces the shutdown pattern to be:

1. **Save via Open API + confirm via `ModelSave` event** — guarantees model bytes are on disk.
2. **After save confirmation, `Process.Kill` is safe** — there is nothing left to lose. View state, undo stack, current selection, dialog positions all get discarded — but those are transient session state, not user work.

In other words: **"clean shutdown" = clean save THEN force-kill.** The save is what makes it clean. The kill is just the mechanism that actually stops the process — because Tekla refuses to do it itself.

### The canonical save-then-kill sequence

```
1. Connect Open API (new Model() + verify GetConnectionStatus())
2. Subscribe to ModelSave event BEFORE triggering save
3. Register events (events.Register())
4. CommitChanges() — flush the modification batch
5. Save() — returns false if model is already clean (no event will fire, that's expected)
6. If Save() returned true → wait for ModelSave event (bounded, typically <5s for blank/small models)
7. UnRegister events
8. Process.Kill() — this is safe because step 5+6 just confirmed the bytes are on disk
9. Process.WaitForExit
```

Optionally subscribe to `TeklaStructuresExit` as well — but be aware it fires **inside Tekla just before process death**, so an external sidecar receives it via Trimble.Remoting just before the channel drops. It's diagnostically useful but not strictly necessary for the shutdown sequence.

### Clean shutdown via Open API + Model events (correct default)

```csharp
using Tekla.Structures.Model;
using Tekla.Structures.Model.Operations;

void ShutdownClean(int pid, int saveTimeoutSeconds = 60, int exitTimeoutSeconds = 60)
{
    // 1. Attach Open API to the target Tekla.
    var model = new Model();
    if (!model.GetConnectionStatus())
        throw new InvalidOperationException("Open API not reachable — can't shut down cleanly");

    // 2. Subscribe to BOTH save and exit signals BEFORE triggering anything.
    using var saveCompleted = new ManualResetEventSlim(false);
    using var exitConfirmed = new ManualResetEventSlim(false);
    var events = new Events();
    events.ModelSave           += () => saveCompleted.Set();
    events.TeklaStructuresExit += () => exitConfirmed.Set();
    events.Register();

    try
    {
        // 3. Flush the modification batch (cheap when nothing pending).
        model.CommitChanges();

        // 4. Trigger save. Returns false if the model is already clean —
        //    that's not an error, just means no ModelSave event will fire.
        bool savedSomething = model.Save();
        if (savedSomething &&
            !saveCompleted.Wait(TimeSpan.FromSeconds(saveTimeoutSeconds)))
        {
            throw new TimeoutException(
                $"ModelSave event did not fire within {saveTimeoutSeconds}s — save still in progress");
        }

        // 5. Trigger app exit. CRITICAL: Process.CloseMainWindow (WM_CLOSE)
        //    is silently ignored by Tekla. Only the Open API call closes it.
        Operation.ExitTekla();

        // 6. Wait for either the TeklaStructuresExit event (inside-process
        //    signal — may arrive late or get cut off by channel drop) OR
        //    Process.HasExited (OS-level signal — reliable backup).
        var p = Process.GetProcessById(pid);
        var sw = System.Diagnostics.Stopwatch.StartNew();
        while (sw.Elapsed.TotalSeconds < exitTimeoutSeconds)
        {
            if (exitConfirmed.IsSet || p.HasExited) return;
            Thread.Sleep(500);
        }
        throw new TimeoutException(
            $"Tekla did not exit within {exitTimeoutSeconds}s of Operation.ExitTekla(). " +
            "Pass force=true if you accept losing any unsaved view state.");
    }
    finally
    {
        events.UnRegister();
    }
}
```

**Three rules embedded in this pattern:**

1. **`Save()` returning `false` is not an error** — it means the model is already clean on disk. Skip the event wait and proceed.
2. **`Process.CloseMainWindow()` is a no-op for Tekla as a whole** — silently ignored. Only `Operation.ExitTekla()` (or its `Exit` / `CloseTekla` variant on older versions — discover the right name via reflection) actually closes the application.
3. **Both signals together, neither alone:** `TeklaStructuresExit` fires inside Tekla; if the remoting channel drops first you'll miss it. `Process.HasExited` is the OS-level backup. Whichever returns first wins.

### Force shutdown — emergency only, with explicit acknowledgement

```csharp
// USE ONLY when: throwaway model / known no unsaved state / Tekla is hung.
// This is destructive. Callers should pass acknowledgeDataLoss explicitly
// so it's never a silent default.
void ShutdownForce(int pid, bool acknowledgeDataLoss)
{
    if (!acknowledgeDataLoss)
        throw new InvalidOperationException(
            "ShutdownForce will lose unsaved model state. Pass acknowledgeDataLoss=true to confirm.");
    var p = Process.GetProcessById(pid);
    p.Kill();
    p.WaitForExit(15_000);
}
```

### Pattern selection

| Scenario | Use |
|---|---|
| Production composition finishing up — model has been edited | **Clean** (always) |
| Test harness `setUp` that just launched a `/create:` blank model and the model has no edits | Force OK |
| Test `tearDown` after an integration test that ran writes against a real model | **Clean** (unless the test explicitly created a throwaway with `/create:`) |
| Tekla hung — process unresponsive, Open API throws RemotingIOException repeatedly | Force OK as last resort, with explicit logging |

### Test matrix simulating "no Tekla running"

For AWARE acceptance test #7 (neither Tekla running), the simulation must be **clean** if the user has any models open with edits, force only when the test owns the models. Pattern:

```csharp
foreach (var p in Process.GetProcessesByName("TeklaStructures"))
{
    try { ShutdownClean(p.Id); }
    catch
    {
        // If clean shutdown fails (e.g. Tekla hung) we choose explicitly
        // to surface the failure rather than silently destroy user data.
        Console.Error.WriteLine($"PID {p.Id}: clean shutdown failed, NOT force-killing automatically.");
        Console.Error.WriteLine("Re-run with --acknowledge-data-loss if you accept the risk.");
        throw;
    }
}
```

There's a ~5-10s grace period before another instance of the same version can launch (Tekla cleans up its IPC MMF artefacts). Wait that out before re-launching.

## Multi-instance limitations — what's possible and what isn't

| Scenario | Possible? | Why |
|---|---|---|
| Same-version, two instances (e.g. two Tekla 2026s) | **No — hard limit** | Tekla refuses to start a second instance and shows "Tekla Structures is already running". Confirmed against Tekla 2025/2026 on standard subscription licenses. Not just a "rare case" — there is no licensing tier where this works. Single-process-per-version is enforced at startup. |
| Cross-version (Tekla 2025 + Tekla 2026 simultaneously) | **Yes** | Different exe + different .NET runtime + different licence token paths. This is what AWARE's v0.29 multi-version routing relies on. |
| Same-version across different Windows user sessions | Yes, but requires session switching — not test-harness friendly, irrelevant to AWARE. |

**Implication for AWARE:** there is no point implementing COM ROT binding for "two same-version Tekla instances with different PIDs". That scenario can't occur. The version-specific DLL load that the `aware-tekla` sidecar already does (loading `Tekla.Structures.Model.dll` from the target Tekla's `Net48Runtime/` folder) is **sufficient** routing for every possible real-world scenario. Don't waste cycles on ROT binding — it would solve a problem that doesn't exist.

## Worked example — full E2E launch + verb + shutdown

```csharp
var bypass = WriteBypassIni(env: "blank_project", role: "Engineer", license: "Partner");
LaunchTeklaHeadless("2026.0", modelPath: @"C:\TeklaStructuresModels\New model 4", bypassIniPath: bypass);
if (!WaitForOpenApiReady("2026.0", timeoutSeconds: 120))
    throw new InvalidOperationException("Tekla 2026 failed to reach Open API readiness within 2 minutes");
// Now safe to invoke Operation.DisplayPrompt, Model verbs, etc.
Tekla.Structures.Model.Operations.Operation.DisplayPrompt("AWARE headless smoke test");
// On teardown:
ShutdownForce(Process.GetProcessesByName("TeklaStructures").First().Id);
```

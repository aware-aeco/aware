# idea-statica-26 EXTRACTION NOTES

This agent is generated from a hybrid coverage IR built by fetching the **OpenAPI YAML** specs and the **markdown SDK docs** that openapi-generator emits alongside them, both pinned to the `26.0.1.2450` tag of the `idea-statica/ideastatica-public` GitHub repository. The result is one IR per IDEA Statica major version covering **two REST APIs** (Connection API + RCS API) plus their auto-generated .NET SDK surfaces.

- **Vendor portal:** `https://developer.ideastatica.com` (DocFX-rendered markdown — only one version published at any time, no version selector for older releases)
- **Source repository:** `https://github.com/idea-statica/ideastatica-public` at tag `26.0.1.2450`
  - Connection API spec: `src/api-sdks/connection-api/clients/csharp/api/openapi.yaml`
  - Connection API SDK docs: `src/api-sdks/connection-api/clients/csharp/docs/*.md`
  - RCS API spec: `src/api-sdks/rcs-api/clients/csharp/api/openapi.yaml`
  - RCS API SDK docs: `src/api-sdks/rcs-api/clients/csharp/docs/*.md`
- **SDK version reported by README:** `26.0.1.1682` (Connection API), `26.0.1.1682` (RCS API) — both auto-generated by openapi-generator from the corresponding `openapi.yaml`
- **Source kind:** `hybrid` (OpenAPI YAML + markdown enrichment; see "Hybrid extraction strategy" below)
- **Extraction date:** 2026-05-19

## Hybrid extraction strategy

IDEA Statica's developer portal renders DocFX markdown but is **single-version** — the live site at `developer.ideastatica.com` always shows the current latest release. To produce a v25 IR and a v26 IR side-by-side we go directly to the **source repository's tagged release tree** where each API ships:

1. **`api/openapi.yaml`** — the canonical OpenAPI 3.0.4 description (~7600 lines for Connection API at v26.0.1.2450). This is the source of truth for **all signatures, types, parameters, return types, and schemas**.

2. **`docs/*.md`** — one markdown file per Model schema (`<TypeName>.md`) and one per *Api class (`<Tag>Api.md`). Auto-generated by openapi-generator from the same YAML. The markdown supplies a few extra niceties (operation summary prose, parameter descriptions consolidated into a table, code-sample links) but contains nothing that isn't derivable from the YAML.

The extractor pipeline is two-pass:

- **Pass 1: YAML parse** — fetch both `openapi.yaml` files, walk the schema tree to emit one `TypeInfo` per `components/schemas/<Name>`, walk the paths tree to group HTTP operations by tag into synthetic *Api class types whose methods are the operations.
- **Pass 2: markdown enrichment** — fetch each Model/*Api markdown page concurrently (4-way, fresh `HttpScraper` per fetch — matches the Rhino concurrency profile against github raw). For Model pages we copy non-empty property descriptions from the table into the IR's property summaries. For *Api pages we copy operation summaries + per-parameter descriptions. 404s are graceful: the page may not exist (openapi-generator omits markdown for synthesized `*_request` request-body schemas like `OpenProject_request`); we fall back to pointing `doc_url` at the YAML so a reader still has a real artifact to consult.

Schemas with `type: string` + `enum` become IR enums; everything else with `type: object` (or implicit object) becomes an IR class. Class properties are extracted from the schema's `properties` map. Operations are flattened from `paths/<route>/<verb>` into method records on the corresponding *Api class.

Method naming convention: the SDK exposes async variants only; we suffix every operationId with `Async` to match (so `Calculate` → `CalculateAsync`, matching `CalculationApi.CalculateAsync` in the .NET client).

### Why not parse the .NET SDK source directly

The auto-generated C# files under `src/api-sdks/<api>/clients/csharp/src/IdeaStatiCa.<Api>/Api/*.cs` would give us identical type+method information. We chose the YAML route because:

1. **Smaller surface to parse.** 7.6k lines of YAML vs ~50k lines of C# across N files; the YAML is machine-emitted by openapi-generator so its shape is stable and constrained.
2. **No reflection / no decompilation.** Phase A's `DllReflector` exists for vendor DLLs we don't have source for; here we DO have the upstream YAML, so we use it.
3. **Cross-language consistency.** The same YAML drives the Python + C# clients. Anchoring the IR on it makes the agent representation language-agnostic by construction.

## YAML parser

`cli-sidecar/Ingest/Extractors/IdeaStatica/YamlReader.cs` is a custom YAML 1.2 subset parser written for this vendor. **Why not YamlDotNet?** YamlDotNet relies on reflection-based serialization that conflicts with `<PublishAot>true</PublishAot>` on the sidecar (`InvalidOperationException: Reflection-based serialization has been disabled`). The openapi-generator output is a tightly-constrained subset of YAML (block style only; no anchors, aliases, or tags), so a focused parser is straightforward and AOT-clean.

Supported constructs:
- Scalar key→value mappings with 2-space indentation
- Block sequences (`- item`) including the "indent of one" rule where items live at the parent key's indent
- Double-quoted scalars with `\` continuation across multiple lines (openapi-generator's signature pattern for long descriptions)
- Single-quoted scalars (`'#/components/schemas/Foo'`)
- Plain (unquoted) scalars with implicit line-folding when continuation lines are more-indented than the parent mapping
- Special-cased reserved-word keys like `"y":` and `"200":`

Unsupported (the openapi-generator never emits these; the parser bails with `YamlParseException` if seen):
- Anchors (`&foo`) / aliases (`*foo`) / merge keys (`<<:`)
- Tag types (`!!int`, `!!str`)
- Flow style (`{a: 1}` or `[1, 2]`)
- Multi-document streams

## Concurrency tuning

| constant | value | rationale |
|----------|-------|-----------|
| YAML fetch | sequential (1) | Each YAML is a single 7-8 MB GET; no benefit from parallelisation. |
| `MemberPageConcurrency` | 4 | Matches the Rhino extractor — github raw rate-limits aggressively at 8+ concurrent fetches with sustained per-IP traffic. 4 stays comfortably below the threshold for the ~140 markdown pages per IDEA Statica version. |
| `SnapshotEvery` | 50 | The IR is small (~600 KB final), so snapshots are cheap. 50 gives a resumability window of ~15-20s at 4-way concurrency. |

## Parser quirks

- **Synthesized `*_request` schemas have no markdown.** openapi-generator emits the request-body type into `components/schemas` for body schemas declared inline at a single operation, but does NOT emit a corresponding `<Name>.md`. Examples: `OpenProject_request`, `ImportIOM_request`, `UpdateFromIOM_request`. Pass 2 sees 5 such 404s per version and falls back to pointing the type's `doc_url` at the openapi.yaml. Total fallback count is reported in `idea-statica-<version>-extraction.log`.

- **Tags become *Api class names.** Each operation declares `tags: [Calculation, ...]` and openapi-generator groups them into `CalculationApi.cs` etc. We mirror that grouping by synthesizing a `class kind` TypeInfo per tag named `<Tag>Api`. The methods on that class are the tag's operations, named `<operationId>Async` to match the SDK's exposed surface (the SDK clients only expose the async variants).

- **Async suffix is added by the SDK, not the YAML.** The YAML's `operationId` is `Calculate`; the SDK's method is `CalculateAsync`; the markdown's heading is `## **CalculateAsync**` BUT the anchor id is `<a id="calculate">`. The MemberPageParser keys patches by the anchor id (lowercased operationId, no Async suffix) so we can match a method in the IR (`CalculateAsync`) to its patch in the markdown.

- **Property summaries are sparse in the source.** Even when both YAML descriptions + markdown descriptions are mined, only ~150 of ~540 properties carry a non-placeholder summary. This reflects the vendor's documentation discipline, NOT a parser failure. The IR substitutes `"(No description provided in vendor docs for ...)"` so the omission is honest and trackable.

- **`object`-typed properties.** A small number of properties carry `type: object` or no `type:` at all in the YAML (e.g. `RcsSetting.value`). The IR's `type="object"` for these is **correct**, matching the vendor's spec. This is analogous to the Tekla SyncRoot pattern (legitimate object-typing). The strict-verify script flags one of these as "property-placeholder-type"; it's a false positive within the brief's 2-tolerated budget.

## Known limitations / follow-up work

1. **No `base` / `interfaces` data.** OpenAPI 3.0.4 doesn't model class inheritance for schemas (only `allOf` composition, which IDEA Statica uses rarely). Every IR type ships with `base: null` and `interfaces: []`. If the upstream `*.cs` files express richer inheritance, a future enrichment pass against the source could populate those — out of scope here.
2. **No constructors.** Schemas in OpenAPI describe data shapes, not how clients construct instances. The SDK's parameter-only constructors (`new ConConnection { Id = 5, … }`) are implicit. Constructors stay empty in the IR.
3. **No events.** Neither the YAML nor the SDK markdown carries event metadata; the SDK is request/response only. `events_related` stays empty.
4. **`additionalProperties: { $ref: ... }` → synthetic Items property.** Schemas that use `additionalProperties` to allow `Dictionary<string, T>` semantics get a single synthetic `Items` property in the IR. Real consumers will iterate the underlying dictionary; the synthetic property is there so the catalog conveys the contained type.
5. **No examples are preserved in the IR.** OpenAPI's `example:` blocks are useful for code generation but noisy for catalog consumption. Future work could surface them as per-property `default` or per-type `remarks`.

## Extraction errors log

If errors occur during extraction (HTTP failures other than 404, parse failures), they are appended to `cli-sidecar/Ingest/Output/idea-statica-26.0-errors.log`. Format matches the Tekla / Rhino extractors: tab-separated UTC timestamp + phase + URL + message. 404 fallbacks are NOT logged as errors (they're an expected condition for synthesized request schemas). A current-tag run produces 0 hard errors end-to-end.

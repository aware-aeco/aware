### Step 2 mismatches

All failures fall into two parser-defect classes — both rooted in the same line of `PageParser.CleanInlineText`:

**P1 — Constructor names drop overload disambiguator (7/50 types).** The vendor's constructor-row text shows distinct overload signatures, e.g. `TransformationPlane()` and `TransformationPlane(CoordinateSystem)`. The catalog stores both as the bare type name (`TransformationPlane` twice), losing the disambiguator. Root cause: `PageParser.CleanInlineText` strips the trailing `<span id="LST...">` substitution which represents the `()` parens, replacing it with a hardcoded `.` or empty. The cs= value in the LST script for those regions is e.g. `cs=()` for the default-constructor empty parens. The parser hardcodes `.` for ALL LST substitutions instead of reading the cs= value.

**P2 — Generic type-parameter brackets in method names mangle to dots (3/50 types).** The vendor's method-row text shows generic types as `IEnumerable<Angle>`. The catalog stores them as `IEnumerable.Angle.` — angle brackets replaced with dots. Same root cause: the LST script for generics carries `cs=&lt;` and `cs=&gt;` (HTML-encoded `<` and `>`), but `CleanInlineText` hardcodes `.` substitution instead of decoding entities and using the cs= value.

Fix (single-file): in `cli-sidecar/Ingest/Extractors/Tekla/PageParser.cs`, change `CleanInlineText` / `CleanNoLinkText` so that when traversing an element with `id` starting with `LST`, the substitution character is read from the next-sibling `<script>` content (the `AddLanguageSpecificTextSet("LST..._N?cs=X|...")` pattern), HTML-entity-decoded, instead of hardcoded `.`.

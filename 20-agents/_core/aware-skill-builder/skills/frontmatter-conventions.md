# AWARE frontmatter conventions

**Two required fields: `name` and `description`. Both have AWARE-specific shape requirements beyond what `skill-creator` enforces.**

## The minimum

```yaml
---
name: <agent>-<short-topic>
description: This skill should be used when <specific trigger contexts>. Encodes <the rule>. Apply when <specific actions>.
---
```

## `name` field — agent-prefixed

Every AWARE skill name starts with its parent agent's id, hyphen-separated from the topic:

| Agent | Skill file | `name` value |
|---|---|---|
| `tekla` | `drawing-identity.md` | `tekla-drawing-identity` |
| `tekla` | `event-threading.md` | `tekla-event-threading` |
| `trimble-connect` | `auth-flow.md` | `trimble-connect-auth-flow` |
| `trimble-connect` | `files.md` | `trimble-connect-files` |
| `revit` | `transactions.md` | `revit-transactions` |

The filename is short (`drawing-identity.md`), the `name` field is long (`tekla-drawing-identity`). The agent prefix makes names globally unique across the AWARE registry — important when an AI ingests skills from multiple agents simultaneously.

Why both? The filename is the contributor-facing label (compact, lives inside an agent folder). The `name` field is the AI-facing identifier (unique, self-describing across agents). Don't conflate.

## `description` field — three required components

A good AWARE skill description has three parts:

1. **Trigger condition** — *"This skill should be used when …"* (third-person, lists specific contexts)
2. **Substance summary** — *"Encodes …"* or *"Covers …"* (what the skill teaches)
3. **Application directive** — *"Apply when …"* (concrete invocation contexts)

### Worked example

```
This skill should be used when authoring AWARE compositions or generated code
that uploads files to Trimble Connect, downloads files, browses or modifies
folders, lists file versions, or links files to BCF topics.

Covers the 3-step package upload pattern (initiate → PUT to pre-signed S3
URL → complete), the 2-step download pattern, folder navigation by path,
file versioning, and the critical conventions (apiBaseUrl already includes
/2.0, IDs are globally unique across projects, DELETE uses versionId not id,
rootId not rootFolderId).
```

That description:

- Opens with the trigger condition in third-person form.
- Enumerates **specific** trigger phrases (upload, download, browse, modify, list versions, link to BCF) so the AI's skill-selection model picks it on those exact words.
- Surfaces the gotchas in the description itself — the AI sees them even before opening the file body.

### Anti-patterns

| Anti-pattern | Why it fails |
|---|---|
| *"Use this skill when …"* | Second-person form; skill-creator and many of Anthropic's published skills use third-person consistently. |
| *"Skill for Trimble Connect file operations."* | Too narrow; no trigger phrases; the AI may not pick it for "upload BIM model to TC". |
| *"Covers everything you need to know about Tekla."* | Too broad; will trigger on irrelevant prompts (false positives). |
| Description over 600 chars | Won't fit comfortably in the AI's skill-selection context window. Target 200–500 chars. |

## Fields that must NOT appear in AWARE skills

These are FloLess-runtime-specific. Strip them on every port:

| Field | Origin | Action |
|---|---|---|
| `group:` | FloLess `BuiltInSkills.cs` runtime — maps the skill to a UI group | **Strip.** AWARE has no UI-group concept. The parent folder is the implicit group. |
| `version:` (on skill itself) | Some early FloLess skills had this | **Strip.** Skill versions are derived from the parent agent's manifest version. |
| `compatibility:` | FloLess WPF runtime compatibility | **Strip.** Out of scope for AWARE substrate. |
| Any `floless-*` prefix in name | FloLess-runtime tagging | **Strip prefix.** Replace with the agent name. |

## Fields that are OPTIONAL but useful

| Field | When to add | Example |
|---|---|---|
| `tier: 1` | Judgment / gotcha skill (the kind contributors hand-write) | For `tekla-drawing-identity` |
| `tier: 2` | API reference skill (generated by `aware agent build`) | For `tekla-structures-model` |
| `examples: [...]` | When the skill is so specific that example triggers help | When the skill name might be ambiguous |

These are AWARE-specific extensions. Anthropic's `skill-creator` won't add them automatically; the AWARE pipeline does in Step 2.

## Verification before commit

After Step 2, the frontmatter must:

- Have `name:` matching `<agent>-<filename-without-extension>`
- Have `description:` starting with *"This skill should be used when"*
- Have no `group:`, `version:`, `compatibility:`, or other FloLess-only fields
- Total description length: 200–500 characters
- Description names at least 3 concrete trigger phrases

If any of those fail, re-invoke `skill-creator` in `optimize-description` mode before proceeding to Step 3.
